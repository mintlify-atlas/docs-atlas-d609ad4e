## Overview

The Cursor UI Clone uses CSS custom properties (CSS variables) defined in the `:root` pseudo-class to create a centralized design token system. All color values are stored as reusable tokens, enabling consistent theming across the application.

## Token Architecture

```css styles.css
:root {
    --color-theme-bg: #14120b;
    --color-theme-fg: #edecec;
    --color-theme-fg-02: #d7d6d5;
    --color-theme-card-hex: #1b1913;
    --color-theme-card-rgb: 27 29 39;
    --color-theme-card-01-hex: #1d1b15;
    --color-theme-card-02-hex: #201e18;
    --color-theme-card-03-hex: #26241e;
    --color-theme-card-04-hex: #2b2923;
    --color-theme-card-hover-hex: #201e18;
    --color-soft: color-mix(in oklab, var(--color-theme-fg)60%, transparent);
}
```

<Note>
**Design System Approach:** All colors are centralized in CSS custom properties, making theme changes as simple as updating values in one location.
</Note>

### Semantic Colors

<Accordion title="Soft Text: --color-soft">

**Value:** `color-mix(in oklab, var(--color-theme-fg)60%, transparent)`

**Dynamic Value:** Computed from `--color-theme-fg` at 60% opacity

**Usage:** Muted/secondary text throughout the UI

```css
.nav__center a:hover {
    color: var(--color-soft);
}

.commenter__details>p:nth-of-type(2) {
    color: var(--color-soft);
}

.footer__end--item1 {
    color: var(--color-soft);
}
```

**Where it's used:**
- Navigation hover states
- Secondary text in testimonials
- Footer text
- Timestamps and metadata
- Descriptive subtitles

**Alternative inline usage:**
```css
.card__item1 h4 {
    color: color-mix(in oklab, var(--color-theme-fg) 60%, transparent);
}
```

<Note>
**Color-mix Function:** Uses the OKLAB color space for perceptually uniform color mixing, creating more natural-looking transparency than traditional rgba.
</Note>

</Accordion>

## Token Naming Convention

### Pattern Analysis

```
--color-[context]-[element]-[variant]-[format]

Examples:
--color-theme-bg                 (theme, background)
--color-theme-fg                 (theme, foreground)
--color-theme-card-01-hex       (theme, card, variant 01, hex format)
--color-theme-card-rgb          (theme, card, rgb format)
```

**Breakdown:**
- `color` - Type prefix
- `theme` - Context (all tokens are theme-related)
- `bg/fg/card` - Element type
- `01/02/03/04` - Variant number (card depth levels)
- `hex/rgb` - Format suffix

## Design Token Benefits

<Accordion title="Centralized Theming">

All color values in one place makes theme creation simple:

```css
/* Dark theme (current) */
:root {
    --color-theme-bg: #14120b;
    --color-theme-fg: #edecec;
}

/* Light theme (example) */
:root {
    --color-theme-bg: #ffffff;
    --color-theme-fg: #14120b;
}
```

No need to search through CSS file - just update `:root` values.

</Accordion>

<Accordion title="Runtime Theme Switching">

CSS custom properties can be changed with JavaScript:

```javascript
// Switch to light theme
document.documentElement.style.setProperty('--color-theme-bg', '#ffffff');
document.documentElement.style.setProperty('--color-theme-fg', '#14120b');
```

This enables dynamic theme switching without reloading CSS.

</Accordion>

<Accordion title="Consistent Design Language">

Using tokens ensures visual consistency:

```css
/* Instead of */
.card-1 { background: #1d1b15; }
.card-2 { background: #1d1b15; }
.card-3 { background: #1c1b14; } /* Oops, slight variation! */

/* Use tokens */
.card-1,
.card-2,
.card-3 {
    background: var(--color-theme-card-01-hex);
}
```

</Accordion>

## Best Practices

<Note>
**✅ Do:**
- Always use `var(--token-name)` instead of hard-coded colors
- Follow the existing naming convention
- Use appropriate card depth levels to create hierarchy
- Use `--color-soft` for secondary text
- Test color contrast for accessibility

**❌ Don't:**
- Hard-code colors (except for truly unique one-off cases)
- Skip token levels (e.g., don't go from bg to card-04)
- Override tokens with `!important`
- Use tokens for non-theme colors (e.g., brand orange)
</Note>
